Technical Decision

왜 이렇게 바꿨는가

4축 → 5축

  • PE 제거 — Hard Filter에서 이미 처리
  • TPO 30% — "상황에 맞는가"가 최우선
  • 리스크 신설 — 감점형(-30~0), 안전 우선
  • 중복 감점 금지 — 축 간 독립 보장

리스트 → Decision-First

  • 10개→1개: 결정 피로 해소
  • 비교/저장/가격비교 과감히 제거
  • Explore는 보조 — 기본은 Top1

PostgreSQL → JSON

  • MVP에서 DB 세팅 = 2주 지연
  • 1,645개 JSON으로 충분히 운용
  • 5축 사전 계산 → 실시간 불필요
  • DB 모델 설계는 유지 (전환 준비)

Reason → 3파트 구조

  • core — 추천 이유 한 문장
  • risk_guard — 왜 안전한 선택인지
  • situation — 언제 입으면 좋은지

Data Collection Pipeline

16만 8천 개에서 1,645개까지

수집 → 정제 → 조합 → 평가 → 검���

네이버 API 167K → 성별 검증 4.8K → 코디 조합 1.6K → LLM 품질 → 최종 1,645개

수집 상세

  • 네이버 쇼핑 API — 12톤 × 카테고리별 크롤링
  • 색상 추출 (PIL + K-means) → 12톤 매핑
  • 하이브리드 분류: 키워드 70% + LLM 캐시 27% + 실시간 3%
  • 성별 재수집: 타이틀 "남성/여성" 키워드 필수

최종 데이터

  • 1,645개 코디 (4,280개 아이템)
  • 192 조합 전부 5개 이상 커버
  • 5축 v2 스코어 사전 계산 포함
  • 4개 규칙 JSON (체형/비율/스타일/격식)

QA & Validation — 핵심

QA Simulator로 발견한 문제들

뷰어 vs 앱 결과 불일치

원인: 뷰어는 v1 스코어, 앱은 v2 실시간 계산
해결: v2 스코어 사전 계산 + _app_rank 동기화

성별 이미지 오염

남성 코디에 여성 상품 이미지 30%+ 혼입
해결: 타이틀 키워드 검증 + 남녀 풀 분리 수집

TPO 스타일 편향

면접 코디 95%가 formal → 캐주얼 선택 시 0건
해결: TPO_STYLE_REINTERPRET (면접+캐주얼→미니멀)

검증 구조

QA Simulator v6

  • 앱과 동일 조건으로 결과 비교
  • API 직접 호출 동기화 모드
  • 아이템 상세 모달 QA (product_id, CTA 유효성)
  • 필터 영향도 단계별 추적

검증 원칙

  • "잘 추천하는 것보다 틀리지 않게 만드는 것"
  • 가격 불일치 → total_price = items 합산 강제
  • 스타일 충돌 → formality 중복 감점 금지
  • 0건 방지 → soft scoring fallback

데이터 품질

상품 이미지 성별 불일치 다수
남녀 풀 분리 (2,187 + 2,005개) + 키워드 검증

속도 vs 정확도

사전 계산 + API 실시간 보정 하이브리드

Learning & Next

배운 것 & 다음 단계

배운 점

  • 추천 정확도 ≠ 사용자 만족 — 결정 구조가 핵심
  • 데이터 품질이 UX를 좌우 — 이미지 성별 오류가 신뢰 붕괴
  • 검증 없는 추천은 위험 — QA Simulator가 구조 개선 주도
  • MVP에서의 기술 선택 — 속도와 구조 사이 균형

현재 완료

  • 전체 흐름 동작 (온보딩→추천→결정)
  • 5축 v2 + Reason v2 + Stylist Criteria
  • iOS 데모 UI + QA Simulator v6

다음 단계
