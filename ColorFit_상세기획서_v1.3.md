## 1. 프로젝트 개요

### 1.1 프로젝트 비전

ColorFit(컬러핏)은 퍼스널컬러 진단 결과와 실제 패션 쇼핑 사이의 단절을 해소하는 AI 패션 의사결정 엔진이다. 기존 서비스들이 "영감"과 "탐색"에 집중하는 반면, ColorFit은 **"왜 이 옷이 나에게 어울리는지"를 논리적으로 설명하고, 더 합리적인 가격의 대안까지 제시**하여 소비자의 결정 피로를 획기적으로 줄인다. "내 색을 아는 순간, 선택이 쉬워진다"는 슬로건 아래, 탐색이 아닌 결정의 종착지를 지향한다.

### 1.2 핵심 목표

| # | 목표 | 설명 | 성공 지표 (KPI) |
|---|------|------|----------------|
| G1 | 온보딩 완주율 | 퍼스널컬러+TPO+예산 5단계 온보딩을 이탈 없이 완료 | 진단 시작 → 완료율 ≥ 60% |
| G2 | 회원 전환 | 온보딩 완료 사용자를 가입 사용자로 전환 | 진단 완료 → 회원가입 전환율 ≥ 40% |
| G3 | 구매 연결 | 추천 코디에서 실제 외부 쇼핑몰로 이동 | 코디 추천 → 외부 링크 클릭율 ≥ 15% |
| G4 | 재방문 유도 | 저장·피드백 루프로 습관적 재방문 생성 | 주간 재방문율 (WAU/MAU) ≥ 30% |
| G5 | 설명 가치 입증 | 추천 이유가 실제 구매 결정에 영향을 줌 | 추천 이유 열람율 ≥ 50% |
| G6 | 추천 신뢰도 | 추천 코디의 퍼스널컬러 적합도 확보 | 전문가 평가 색상 매칭 정확도 ≥ 80% |
| G7 | 추천 코디 품질 | 코디 피드의 save 전환율 확보 | save율 ≥ 20% |
| G8 | LLM 품질 보증 | Gemini 평가 기반 코디 품질 확보 | LLM 4점 이상 코디 비율 ≥ 60% |
| G9 | 스타일 적합도 | 규칙 기반 스타일 필터 통과율 | StyleFilter 55점+ 통과율 ≥ 70% |

### 1.3 프로젝트 범위

**제품 기능 범위**

**Must — 핵심 기능**
- **성별 선택** (남성/여성) — 온보딩 최초 진입 시 선택. TPO·무드·카테고리 세트가 성별에 따라 분화
- 퍼스널컬러 12-tone 선택 온보딩 (직접 선택 방식)
- TPO 8종(성별별 상이) + 스타일 무드(성별별 상이) + 예산 설정
- 개인화 코디 피드 (스코어링 기반 정렬)
- 코디 카드 1줄 요약 + 추천 이유 2줄
- 점수 뱃지 (personalColorFit / occasionFit)
- 아이템 단품 상세 + 판매처별 가격 비교
- Exact / Similar 구분 + 유사 상품 목록
- 최저가 외부 쇼핑몰 링크
- save(저장) / dislike(싫어요) + 피드 즉시 반영
- Top Pick 단일 추천 / A vs B 비교 판정
- 소셜 로그인 (카카오, 구글) + 게스트 탐색
- **Style Fit 5축 스코어링** — 기존 4축(PCF/OF/CH/PE)에 SF축 추가, 가중치 재조정
- **규칙 기반 StyleFilter** — 카테고리 궁합 + 실루엣 밸런스 + 포멀도 일관성 사전 필터
- **Gemini 멀티모달 스타일 스코어러** — 최종 후보 20개 AI 스타일 평가
- **사용자 피드백 개인화 학습** — PreferenceTracker를 통한 가중치 동적 조정
- **TPO 동의어 확장** — commute↔office, weekend↔casual 등 자동 매핑
- **12톤별 병렬 데이터 수집 파이프라인** — 트렌드 리서치 → 톤별 수집 → 전처리 → LLM 평가
- **에디토리얼 매거진 디자인 시스템** — Primary #222222, Accent Marsala #964F4C (섹션 8.6 참조)

**Should — 확장 기능**
- AI 셀카 기반 퍼스널컬러 자동 진단
- 날씨 연동 추천 / 옷장 관리 / 풀세팅 모드
- 커뮤니티 / SNS 공유 카드
- 푸시 알림 (가격 알림, 아침 코디 알림)
- 무신사 / W컨셉 크롤링 수집 (데이터 소스 확장)

**Could — 장기 비전**
- 실제 결제(PG 연동) — 현재 외부 링크 방식으로 대체
- 자체 상품 재고 관리

### 1.4 용어 정의

| 용어 | 정의 |
|------|------|
| 퍼스널컬러 12-tone | 봄/여름/가을/겨울 × 라이트/뮤트/딥 등 12가지 세부 톤 분류 체계 |
| TPO | Time(시간)·Place(장소)·Occasion(상황)의 약자. 코디의 맥락을 정의 |
| personalColorFit | 코디 색상이 사용자의 퍼스널컬러 팔레트에 얼마나 적합한지를 0~100으로 수치화 |
| occasionFit | 코디가 선택한 TPO(출근, 소개팅 등)에 얼마나 적합한지를 0~100으로 수치화 |
| colorHarmony | 코디 내 아이템 간 색상 조화도. 60-30-10 법칙 기반 평가 |
| priceEfficiency | 사용자 설정 예산 대비 코디 총액의 효율성 점수 |
| isCompleteOutfit | 상의+하의(또는 원피스) 이상 구성되어 단독 착용 가능한 완성 코디 여부 |
| Exact Match | 동일 브랜드·동일 상품을 다른 판매처에서 판매하는 경우 |
| Similar Match | 동일 카테고리·유사 색상군·유사 가격대의 대체 가능 상품 |
| Top Pick | 저장 목록 또는 추천 목록 중 종합 적합도 최고 1개를 강조 추천하는 기능 |
| 코디 피드 | 사용자 프로필에 맞춰 개인화 정렬된 코디 카드 리스트 |
| 추천 이유 (reasons) | 코디가 추천된 근거를 자연어로 설명하는 1~2줄 문장 |
| styleFit (SF) | 코디 내 아이템 간 스타일 조화도. 카테고리 궁합·실루엣 밸런스·포멀도 일관성 기반 평가 (0~100) |
| StyleFilter | 규칙 기반 스타일 호환성 사전 필터. 카테고리 궁합(50%) + 실루엣 조화(25%) + 포멀도 일관성(25%) 합산 55점 미만 자동 필터링 |
| GeminiStyleScorer | Gemini 멀티모달 AI 기반 스타일 평가 모듈. 최종 후보 20개에만 적용하여 비용 최적화 |
| PreferenceTracker | 사용자 피드백(save/like/click/dislike)을 학습하여 개인화 가중치를 동적으로 조정하는 모듈 |
| TPO 동의어 확장 | commute↔office, weekend↔casual 등 유사 TPO를 자동 확장하여 매칭 정확도를 높이는 로직 |
| LLM 품질 평가 | Gemini Flash 배치 평가로 코디 품질을 1~5점으로 평가, 3점 이상만 서빙 |
| 성별 프로필 (gender) | 온보딩 Step 1에서 선택. TPO·무드 옵션 세트, 상품 필터, 코디 레시피가 성별에 따라 분화 |
| designed_tpo | 코디 생성 시점에 의도적으로 설정된 TPO (키워드 역산출이 아닌 레시피 기반 설계값) |
| 코디 레시피 (Outfit Recipe) | 성별 + TPO + 무드 + 필수/선택/금지 카테고리 + 포멀도 범위로 구성된 코디 생성 규격 |
| Style Seed (취향 시드) | 온보딩 Step 5 비주얼 취향 분석의 결과. 무드·실루엣·색감·가격 선호를 4라운드 이미지 선택으로 파악하여 초기 개인화에 활용 |
| 취향 초기화 | Style Seed + PreferenceTracker 데이터를 삭제하고 기본 가중치로 복원하는 기능. 마이페이지에서 실행 |

## 2. 문제 정의 및 기획 동기

### 2.1 패션 소비자의 세 가지 반복 고민

| 고민 | 핵심 질문 | 상세 |
|------|----------|------|
| **Fit & Harmony** | "이 옷이 나한테 어울릴까?" | 퍼스널컬러 진단을 받았지만, 막상 쇼핑 시 어떤 색상의 어떤 아이템을 골라야 하는지 연결되지 않음 |
| **Logical Curation** | "왜 나에게 어울리는데?" | 구매를 망설이는 순간, 소비자가 원하는 것은 감각의 강요가 아닌 납득할 수 있는 스타일링 논리 |
| **Smart Consumption** | "동일한 무드, 더 합리적인 대안은?" | 취향에 맞는 스타일을 찾아도 예산에 맞는 대안을 찾기 위해 여러 쇼핑몰을 돌아다니며 시간 소비 |

### 2.2 현재 시장의 한계

| 서비스 | 핵심 강점 | 빠진 고리 (Missing Link) |
|--------|----------|------------------------|
| **Lyst** | 1.6억 유저, 2.7만+ 브랜드의 압도적 탐색 풀 | 개인의 맥락(퍼스널컬러, 체형)과 추천 논리 부재 |
| **Stitch Fix** | 고도화된 1:1 개인화 스타일링 | 폐쇄적 큐레이션, 실시간 가격 비교 불가, 한정된 브랜드 |
| **LTK** | 막강한 크리에이터 기반 룩 발견 | 인플루언서 의존적, 유저 맞춤형 논리 부족 |
| **ModeSens** | 강력한 옴니채널 가격 비교 | 코디 제안 및 초개인화 스타일링 부재 |
| **무신사 / W컨셉** | 국내 최대 패션 커머스 | 퍼스널컬러 기반 필터 없음, 추천 이유 없음 |

### 2.3 White Space

```
                  초개인화(Fit)
                      │
         Stitch Fix   │   ★ ColorFit ★
                      │
  ──────────────────────────────────── 가격 효율(Price)
                      │
         LTK / Lyst   │   ModeSens
                      │
              설명 가능성(Why)
```

**ColorFit이 채우는 빈자리:** 초개인화(Fit) + 납득 가능한 큐레이션(Why) + 가격 효율(Price)을 하나의 여정(Seamless Journey)으로 완결 짓는 통합 결정 엔진

### 2.4 핵심 가설

| # | 가설 | 검증 방법 | 성공 기준 |
|---|------|----------|----------|
| H1 | 퍼스널컬러 기반 추천은 일반 추천 대비 클릭율이 높다 | A/B 테스트: 톤 필터 on/off | CTR 차이 ≥ 20% |
| H2 | 추천 이유 제공은 구매 전환율을 높인다 | A/B 테스트: 이유 표시 on/off | 외부 링크 클릭율 차이 ≥ 15% |
| H3 | 유사 상품 대체재는 이탈율을 낮춘다 | 코디 상세 → 이탈율 비교 | 이탈율 감소 ≥ 10% |
| H4 | 코디 단위 추천은 단품 대비 객단가를 높인다 | 코디 클릭 vs 단품 클릭 평균 가격 | 평균 클릭 코디 가격 ≥ 1.5배 |

## 3. 페르소나 및 사용자 여정 상세

### 3.1 Core 페르소나

#### A. 진단러 — 김지은 (25세, 사회초년생)

| 항목 | 내용 |
|------|------|
| 퍼스널컬러 | 여름쿨소프트 (전문 매장에서 진단 받음) |
| 핵심 불편 | 진단 받았는데 막상 쇼핑할 때 뭘 골라야 할지 모름 |
| 목표 | 진단 결과를 실제 쇼핑에 연결하고 싶다 |
| 주요 TPO | 출근룩, 소개팅, 주말 캐주얼 |
| 예산 | 3~10만원/아이템 |
| 디바이스 | 모바일 (인스타 즐겨봄) |

**시나리오 1: 소개팅 룩 찾기**
> "내일 소개팅인데 여름쿨 톤에 데이트룩으로 뭐 입어야 하지?"
1. ColorFit 앱 실행 → 여름쿨소프트 프로필 로드
2. TPO 탭에서 "데이트" 선택
3. 코디 피드에서 여름쿨 팔레트 기반 코디 노출
4. 추천 이유 확인: "여름쿨소프트 핵심 컬러 라벤더 블루 계열 + 데이트 룩에 적합한 A라인 실루엣"
5. 아이템별 가격 확인 → 최저가 무신사 링크로 이동 → 구매

**시나리오 2: 진단 결과가 뭔지 헷갈릴 때**
> "여름쿨소프트가 정확히 어떤 색이지?"
1. 마이페이지 → 톤 설명 화면 진입
2. 대표 색상 스와치 + "차갑고 부드러운 파스텔 계열이 피부를 밝게 만들어요" 설명
3. 추천색(라벤더, 소프트 핑크) / 비추천색(오렌지, 카키) 팔레트 프리뷰
4. 잘못 골랐다면 → 온보딩 수정으로 재선택

#### B. 바쁜이 — 박민준 (32세, 직장 3년차)

| 항목 | 내용 |
|------|------|
| 퍼스널컬러 | 가을웜딥 (진단 받았지만 기억 희미) |
| 핵심 불편 | 코디 고민할 시간이 없음. 사도 후회할까봐 결정 못함 |
| 목표 | 빠르게 "이거 사도 되겠다" 확신 얻고 싶다 |
| 주요 TPO | 출근룩, 중요한 미팅, 주말 편한 외출 |
| 예산 | 5~15만원/아이템 |
| 디바이스 | 모바일 (출퇴근 중 사용) |

**시나리오 1: 출퇴근 5분 코디 탐색**
> "5분 안에 이번 주 입을 출근룩 후보 3개만 찾고 싶어"
1. 홈 진입 즉시 코디 피드 (로딩 최소화)
2. TPO 탭 "출근" 선택 → 가을웜딥 코디 빠르게 스캔
3. 마음에 드는 2개 저장 → dislike로 나머지 빠르게 넘김
4. 퇴근 후 저장 목록에서 상세 확인 → 구매 결정

**시나리오 2: 미팅 전날 급하게 코디 확정**
> "내일 중요한 미팅인데 오피스룩 빨리 골라야 해, 20만원 이내"
1. 예산 슬라이더를 20만원으로 조정
2. TPO = "오피스/비즈니스" 전환
3. isCompleteOutfit 우선 → 상하의 완성 코디 먼저 노출
4. 코디 전체 최저가 합산 확인 → "총 17.8만원" → 즉시 결정

**시나리오 3: A vs B 비교 판정**
> "두 코디 중 어떤 게 나은지 모르겠어"
1. 저장한 2개 코디를 A vs B 비교 모드로 진입
2. 색상 적합도 / TPO 적합도 / 가격 효율을 나란히 비교
3. "A안이 가을웜딥 적합도 92점, B안은 78점" → A안 결정

**시나리오 1: 마음에 드는 코디의 저렴한 버전**

| 단계 | 목적 | 입력 | 출력 | 핵심 로직 |
|------|------|------|------|----------|
| 1. Trend Research | 시즌별 트렌드 키워드를 도출하여 수집 쿼리 품질 향상 | 패션 트렌드 소스 (매거진, SNS) | 12톤별 트렌드 키워드 리스트 | LLM 기반 트렌드 분석 → 톤별 검색 쿼리 생성 |
| 2. Curate by Tone | 12톤 각각에 최적화된 상품을 네이버 쇼핑 API로 수집 | 톤별 키워드 리스트 | 톤별 raw 상품 JSON | 4개 톤 병렬 수집 × 3라운드, API Rate Limit 준수 |
| 3. Rebuild from Tones | raw 상품을 정규화하고 **TPO×무드 의도 기반** 코디를 설계 생성 | raw 상품 JSON + TPO/무드 레시피 | 정규화 상품 DB + 코디 조합 | HTML 태그 제거, 카테고리 분류, 색상 추출, 톤 매핑, **레시피 기반 코디 조합 생성 (5.3.1 참조)** |
| 4. Evaluate Outfits | LLM 멀티모달 평가로 저품질 코디를 사전 제거 | 코디 조합 리스트 | 품질 통과 코디 (3점 이상) | Gemini Flash 배치 평가, 5점 척도 |
| 피드 서빙 | 사용자 프로필 기반 실시간 5축 스코어링 및 피드 구성 | 통과 코디 + 사용자 프로필 | 개인화된 코디 피드 | 5축 스코어링 + 리랭킹 + 이유 생성 |

**설계 근거:** 기존의 Collect → Clean → Structure → Index → Serve 순차 파이프라인은 수집과 품질 평가가 분리되어 저품질 데이터가 서빙 단계까지 흘러가는 문제가 있다. 트렌드 리서치 단계를 최상단에 배치하여 수집 시점부터 품질을 높이고, LLM 평가 단계를 서빙 전에 게이트로 배치하여 최소 품질 기준을 보장한다.

### 5.2 데이터 소스

네이버 쇼핑 API를 주 수집 소스로 사용한다. 무신사/W컨셉 크롤링은 법적 리스크 검토 후 추가한다.

#### 네이버 쇼핑 API (주 수집 소스)

| 항목 | 내용 |
|------|------|
| 엔드포인트 | `https://openapi.naver.com/v1/search/shop.json` |
| 호출 제한 | 하루 25,000회 (개인 키 기준) |
| 핵심 필드 | `title`, `link`, `image`, `lprice`, `hprice`, `mallName`, `category1~4` |
| 목표 수집량 | **25,000개 이상** |
| 수집 방식 | 12톤별 병렬 수집 — 4개 톤 동시 수집 × 3라운드로 12톤 커버 |
| 브랜드 필터 | 화이트리스트 약 120개 이상 인지도 브랜드 |
| 성별 필터 | 남성 전용 키워드("남성용", "맨즈") 자동 제외 |

**네이버 쇼핑 API를 단일 소스로 선택한 근거:**

1. **공식 API 제공** — robots.txt 위반이나 크롤링 법적 리스크 없이 안정적 수집 가능
2. **충분한 상품 볼륨** — 일 25,000회 호출로 25,000개 이상 상품 확보 가능
3. **구조화된 응답** — 가격, 카테고리, 이미지 URL 등 정규화에 필요한 필드를 표준 형식으로 제공
4. **다양한 판매처** — 무신사, W컨셉, 29CM 등 주요 패션 쇼핑몰 상품이 네이버 쇼핑에 등록되어 있어 사실상 멀티소스 효과

**수집 쿼리 설계:**

```
톤별 트렌드 키워드를 조합하여 검색 쿼리를 생성한다.

TONE_QUERIES 구조:
  톤 ID → [키워드1, 키워드2, ...]

예시:
  "spring_warm_light" → ["봄 코랄 블라우스", "아이보리 원피스", ...]
  "summer_cool_soft"  → ["라벤더 니트", "파스텔 블루 슬랙스", ...]
  ...
  12개 톤 × 카테고리별 키워드 = 전체 쿼리 풀
```

**카테고리별 목표 수집 비율:**

| 카테고리 | 목표 비율 | 근거 |
|---------|----------|------|
| outer | ~19% | 코디 완성도에 가장 큰 영향, 가격대 높아 비교 가치 큼 |
| top | ~19% | 가장 자주 교체하는 아이템, 색상 다양성 핵심 |
| bag | ~13% | 코디 포인트 아이템, 색상 매칭 중요도 높음 |
| shoes | ~13% | 전체 실루엣 완성, TPO 판단 핵심 지표 |
| onepiece | ~13% | 단독 코디 가능, 톤 매칭 정확도 높음 |
| bottom | ~13% | 상의와의 조합 다양성 확보 필요 |
| acc | ~9% | 포인트 아이템, 수집 난이도 대비 비율 조정 |

#### 무신사 / W컨셉 상품 데이터 (추가 수집 소스)

| 항목 | 내용 |
|------|------|
| 접근 방식 | 공개 API 없음 → 상품 페이지 정적 크롤링 + 카테고리 탐색 |
| 핵심 필드 | 상품명, 가격, 브랜드, 카테고리, 색상, 이미지 URL |

| 주의사항 | robots.txt 준수, 요청 간격 2초 이상, 상업적 이용 제한 확인 |

#### 퍼스널컬러 12-tone 팔레트 DB

| 항목 | 내용 |
|------|------|
| 데이터 구성 | 12개 톤 × 각 20~30개 대표 색상 = 약 300개 색상 코드 |
| 형식 | HEX + RGB + HSL + 색상명(한글) |
| 출처 | 퍼스널컬러 전문가 자료 + 색채학 표준 팔레트 |

#### 코디 조합 데이터

코디 조합은 **TPO×무드 레시피 기반 의도적 설계 생성** 방식을 사용한다. 상세 설계는 5.3.1 참조.

| 항목 | 내용 |
|------|------|
| 생성 방식 | **TPO별 레시피(카테고리 구성 + 포멀도 범위 + 무드 키워드)에 따라 의도적으로 조합** |
| 목표 규모 | 2(성별) × 12톤 × 8 TPO × 약 8~10코디 = **1,500~1,900개** 코디 세트 |
| 라벨링 | 카테고리, 색상 태그, **설계된 TPO 태그**, **무드 태그**, 시즌 태그 |
| 품질 보장 | 레시피 적합성 검증 + Gemini Flash 배치 평가 → 3점 미만 제외 |

### 5.3 수집 전략

#### 톤별 병렬 수집 설계

**왜 톤별 병렬 수집인가:**

1. **데이터 균형 보장** — 순차 수집 시 인기 톤(spring_warm_light 등)에 데이터가 편중된다. 톤별로 독립 수집하여 12톤 모두 최소 기준량을 확보한다.
2. **수집 시간 단축** — 12톤을 순차 수집하면 API Rate Limit(25,000회/일)에 의해 최소 3일이 필요하다. 4개 톤을 동시에 수집(tmux 4 세션)하면 3라운드(4+4+4)로 1~2일 내 완료 가능하다.
3. **재시도 용이성** — 톤 단위로 수집이 독립적이므로, 특정 톤의 수집이 실패해도 해당 톤만 재시도하면 된다.

**수집 계획:**

| 항목 | 목표 |
|------|------|
| 상품 수집 | 25,000개 이상 (12톤 × 카테고리별) |
| 코디 생성 | 1,500~1,900개 (레시피 기반 조합) |
| LLM 품질 평가 | 생성된 전체 코디 배치 평가 |
| 12-tone 팔레트 | 12톤 × ~25색 (JSON) |
| 소요 시간 | ~4~5일 |

**톤별 데이터 균형 관리:**

각 톤의 최소 목표치를 1,000개 상품으로 설정한다. 수집 후 톤별 편차를 점검하여, 부족 톤(목표 미달)은 키워드를 추가 확장하여 보완 수집한다. 특히 mute/light 계열 톤은 쇼핑몰에서 해당 키워드의 상품이 적을 수 있으므로, 유사 색상명 동의어(예: "뮤트 베이지" → "그레이쉬 베이지", "먹색")를 활용한 확장 쿼리를 설계한다.

| 톤 그룹 | 수집 난이도 | 대응 전략 |
|---------|-----------|----------|
| spring_warm_light/bright | 낮음 | 표준 키워드로 충분 |
| summer_cool_soft | 낮음 | 파스텔 계열 키워드 풍부 |
| autumn_warm_deep | 중간 | 어스톤 계열 키워드 확장 |
| winter_cool_deep/bright | 낮음 | 블랙/네이비/비비드 키워드 풍부 |
| mute 계열 (spring/summer/autumn) | **높음** | 동의어 확장, 색상 코드 기반 검색, 추가 수집으로 보완 |
| light 계열 (summer/winter) | **높음** | 파스텔/아이시 키워드 확장, 추가 수집으로 보완 |

#### 5.3.1 코디 조합 설계: TPO×무드 레시피 기반 생성

코디 조합은 기존의 랜덤 조합(top×bottom 셔플)이 아닌 **TPO×무드 레시피 기반 의도적 설계 생성** 방식을 사용한다.

**랜덤 조합 방식의 문제:**

| 문제 | 현상 | 원인 |
|------|------|------|
| TPO 역산출 | 코디의 TPO 태그가 아이템 키워드 합산으로 결정됨 | 코디 생성 시 "이 코디는 면접용"이라는 의도 없음 |
| 무드 부재 | 온보딩에서 "클래식/미니멀/러블리" 수집하지만 코디 조합에 미반영 | 무드 개념 자체가 조합 로직에 없음 |
| 부조화 대량 생성 | 후드+정장 스커트, 스팽글 탑+카고 팬츠 등 | 카테고리 궁합 무시한 랜덤 셔플 |
| OF 분포 불규칙 | 면접 사용자에게 casual 태그 코디가 대량 노출 | TPO가 의도가 아닌 결과물이라 분포 제어 불가 |

**레시피 기반 설계:**

코디 생성을 "레시피"로 정의한다. 레시피는 **성별 + TPO + 무드 + 카테고리 구성 + 포멀도 범위 + 실루엣 권장**을 명시한다.

레시피를 성별로 분화한다. 남성과 여성은 동일 TPO(예: 면접)라도 카테고리 구성, 무드 해석, 아이템 풀이 근본적으로 다르다.

```
코디 레시피 (Outfit Recipe)
├── 성별: female
├── TPO: interview
├── 무드: classic, minimal
├── 카테고리 구성:
│   ├── 필수: [셔츠|블라우스] + [슬랙스|스커트]
│   ├── 선택: [블레이저|자켓] + [로퍼|힐]
│   └── 금지: [후드, 맨투맨, 스니커즈, 크롭탑]
├── 포멀도 범위: 3~5
├── 실루엣 권장: slim, fitted, regular (oversized 제외)
└── 가격대 참고: 5~30만원
```

**여성 TPO별 레시피 매트릭스:**

| TPO | 필수 카테고리 | 선택 카테고리 | 금지 | 포멀도 | 무드 |
|-----|-------------|-------------|------|--------|------|
| interview | 셔츠/블라우스 + 슬랙스/스커트 | 블레이저, 로퍼, 힐 | 후드, 맨투맨, 스니커즈, 크롭탑 | 3~5 | classic, minimal |
| commute | 셔츠/블라우스/니트 + 슬랙스 | 자켓, 로퍼, 토트백 | 숏팬츠, 슬리퍼, 레깅스 | 3~4 | classic, minimal |
| date | 니트/블라우스/원피스 + 스커트/슬랙스 | 카디건, 힐, 크로스백 | 패딩, 조거팬츠 | 2~4 | lovely, casual, editorial |
| weekend | 티셔츠/맨투맨/니트 + 청바지/와이드팬츠 | 가디건, 스니커즈 | 블레이저, 힐 | 1~3 | casual, street |

| event | 블라우스/원피스 + 스커트/슬랙스 | 코트, 힐, 클러치 | 후드, 스니커즈, 백팩 | 4~5 | classic, editorial |

| 댄디 | 남 전용 | slim, fitted | 네이비, 버건디, 뉴트럴 | 성숙, 격조 |
| casual | 공통 | regular, oversized | 자유로운 배색 | 편안함, 레이어드 |
| street | 공통 | oversized, wide | 대비, 비비드 | 과감, 믹스 |
| editorial | 여 전용 | fitted, slim | 대비 + 톤온톤 | 구조적, 실험적 |
| 아메카지 | 남 전용 | regular, wide | 어스톤, 인디고, 카키 | 빈티지, 워크웨어 |

**조합 알고리즘 (의사코드):**

```
for each gender in [female, male]:
  for each tone in 12_tones:
    recipes = RECIPES[gender]  ← 성별에 따라 다른 레시피 세트
    for each recipe in recipes:
      target_count = 8~10 per gender (총 16~20 per tone×tpo)
      generated = 0

      while generated < target_count:
        1. 해당 성별 상품 풀에서 필수 카테고리 아이템 선택 (톤 호환 + 포멀도)
        2. 선택 카테고리에서 확률적 추가 (60% 신발, 25% 아우터)
        3. 금지 카테고리 검증
        4. 포멀도 범위 검증 (max - min ≤ 2)
        5. 무드 적합성 검증 (성별별 실루엣·색감 필터)
        6. 가격 비율 검증 (5배 이내)
        7. 중복 조합 검증

        outfit.gender = gender
        outfit.designed_tpo = recipe.tpo
        outfit.designed_moods = recipe.moods
        generated += 1

목표 규모: 2(성별) × 12(톤) × 8(TPO) × 8~10(코디) = 1,500~1,900개
```

**기대 효과:**
- OF(occasionFit) 점수 분포가 TPO별로 균등해져 피드 품질 향상
- 면접 사용자에게 캐주얼 코디가 노출되는 문제 근본 해결
- "클래식" 무드 선호 사용자에게 oversized+wide 조합이 배제됨
- StyleFilter의 SF 축과 시너지 — 생성 단계에서 이미 포멀도/실루엣이 정합하므로 SF 점수 자연 향상

### 5.4 전처리 과정

전처리는 4단계 필터 체인으로 구성한다. 각 필터는 독립적으로 동작하며, 순서대로 적용하여 점진적으로 데이터 품질을 높인다.

#### 상품 데이터 정규화

API 응답의 raw 데이터를 표준화된 내부 포맷으로 변환한다.

```
정규화 프로세스:
  Input:  네이버 쇼핑 API raw JSON
  Output: NormalizedProduct {
            product_id, name, brand, category,
            color_hex, tone_id, price,
            mall_name, mall_url, image_url, tags
          }

  처리 단계:
    1. HTML 태그 제거 (title에 <b> 등 포함)
    2. 브랜드명 추출 (title 파싱 또는 mallName 기반)
    3. 카테고리 분류 → 하이브리드 분류 체계 (5.4.1 참조)
    4. 대표 색상 추출 (이미지에서 dominant color)
    5. 12-tone 매핑 (HEX → 가장 가까운 톤)
    6. TPO/시즌/스타일 태그 생성 → 하이브리드 분류 체계 (5.4.1 참조)
```

#### 5.4.1 하이브리드 분류 체계: 키워드 + LLM 폴백

키워드 기반 분류의 한계(실패율 ~30%)를 해결하기 위해, 키워드 매칭을 1차로 시도하고 실패 시 Gemini 배치 분류로 폴백하는 2단계 하이브리드 구조를 도입한다.

**키워드 기반 분류의 측정된 한계:**

| 사례 | 상품명 | 기대 카테고리 | 키워드 결과 | 원인 |
|------|--------|-------------|-----------|------|
| 미감지 | "워셔블 캐시미어 V넥 풀오버" | 니트 | None | "니트" 키워드 없음 |
| 미감지 | "여성 슬림핏 테일러드 팬츠" | 슬랙스 | None | "슬랙스" 키워드 없음 |
| 오분류 | "린넨 셔츠 원피스" | 원피스 | 셔츠 | "셔츠"가 먼저 매칭 |
| 미감지 | "캐주얼 루즈핏 면 바지" | 바지 | None | 일반 명사만 사용 |

키워드는 **등록된 동의어 범위 내에서만 작동**하며, 브랜드마다 다른 상품명 표기("풀오버", "스웨터", "저지 탑" 등)를 모두 커버할 수 없다.

**하이브리드 분류 설계:**

```
상품명 + 카테고리 힌트
        │
        ▼
  ┌──────────────┐
  │ 1단계: 키워드  │ ← 31개 카테고리 × 3~5개 키워드 = ~120개 규칙
  │ 매칭 (< 1ms)  │    비용: 0원, 속도: 즉시
  └───────┬──────┘
          │
     매칭 성공? ──── Yes ──→ 결과 사용 (~70%)
          │
          No (~30%)
          │
          ▼
  ┌──────────────┐
  │ 2단계: LLM    │ ← Gemini Flash 배치 분류
  │ 캐시 조회     │    사전 분류 결과를 JSON에 캐싱
  │ (< 1ms)      │    비용: 1회 배치 ~$2.5 (25K 상품 × 30%)
  └───────┬──────┘
          │
     캐시 히트? ──── Yes ──→ 결과 사용
          │
          No (신규 상품)
          │
          ▼
  ┌──────────────┐
  │ 3단계: LLM    │ ← 실시간 Gemini 호출 (극소수)
  │ 실시간 분류   │    결과를 즉시 캐시에 추가
  │ (~300ms)     │
  └──────────────┘
```

**LLM 분류 시 요청하는 속성:**

| 속성 | 선택지 | 용도 |
|------|--------|------|
| category | 31종 (니트, 셔츠, 블라우스, ...) | StyleFilter 카테고리 궁합 |
| silhouette | oversized, slim, fitted, wide, regular | 실루엣 조화 평가 |
| formality | 1~5 (캐주얼~포멀) | 포멀도 일관성 |
| tpo | office, date, casual 등 (복수) | TPO 적합도 평가 |
| **gender** | **female, male, unisex** | **성별 매칭 필터 (2단계)** |

**왜 RAG/온톨로지가 아닌 LLM 폴백인가:**

| 대안 | 판정 | 근거 |
|------|------|------|
| 패션 온톨로지 | 불필요 | "더 잘 정리된 수동 규칙"에 불과. 신조어/신규 아이템은 여전히 수동 추가 필요 |
| RAG (벡터DB) | 오버엔지니어링 | 25K 상품 카테고리 분류에 벡터DB 인프라는 문제-해결책 불일치. Railway 메모리 제한에서 인메모리 DB 부담 |
| **LLM 배치 폴백** | **적정** | 이미 Gemini 인프라 보유. 추가 인프라 0. 배치 1회 비용 ~$2.5. 실패율 30%→3% |

**TPO 동의어 확장도 동일 전략 적용:**

기존 하드코딩된 `TPO_SYNONYMS` 딕셔너리 대신, LLM 배치 분류 시 각 상품에 TPO 태그를 직접 부여한다. "하객 원피스" → `["wedding"]`, "캠퍼스 백팩" → `["campus", "casual"]` 등 맥락 기반 판단을 LLM이 수행한다.

#### 필터 아키텍처: Hard Filter + Soft Score 분리

추천 파이프라인의 필터를 **Hard Filter(탈락)** 와 **Soft Score(순위)** 로 명확히 분리한다. Hard Filter를 통과하지 못한 코디는 리스트에서 **완전 제거**되며, Soft Score는 순위만 변경하고 제거하지 않는다.

```
전체 코디 풀
      │
      ▼
  ┌─────────────────────────────────────┐
  │         Hard Filter (탈락 기준)       │  ← "이건 틀리면 안 됨"
  │  통과 못하면 리스트에서 완전 제거       │
  │                                      │
  │  H1. 성별 불일치                      │
  │  H2. 예산 초과 (50%+ 초과)            │
  │  H3. 계절 완전 불일치                  │
  │  H4. TPO 완전 불일치                  │
  │  H5. 브랜드 화이트리스트 미달           │
  │  H6. LLM 품질 미달 (3점 미만)          │
  │  H7. 톤 호환성 0매칭                   │
  │  H8. StyleFilter 컷오프 미달 (55점)    │
  └───────────────┬─────────────────────┘
                  │ 통과한 코디만
                  ▼
  ┌─────────────────────────────────────┐
  │         Soft Score (순위 기준)        │  ← "더 잘 맞을수록 위로"
  │  제거하지 않고 순위만 조정              │
  │                                      │
  │  S1. 퍼스널컬러 적합도 (PCF)           │
  │  S2. TPO 적합도 (OF)                  │
  │  S3. 색상 조화 (CH)                   │
  │  S4. 가격 효율 (PE)                   │
  │  S5. 스타일 적합도 (SF)               │
  │  S6. 개인화 보정 (피드백 학습)          │
  │  S7. Gemini AI 스타일 평가 (선택적)    │
  └───────────────┬─────────────────────┘
                  │ 점수 순 정렬
                  ▼
              코디 피드
```

**설계 원칙:** Hard Filter는 "명백히 잘못된 추천"을 방지한다. 예산을 50% 넘는 코디, 면접에 트레이닝복, 여름에 패딩은 아무리 색상이 잘 맞아도 추천하면 안 된다. 반면 Soft Score의 축은 "얼마나 더 잘 맞는가"를 나타내며, 낮은 점수라도 제거 사유가 되지 않는다.

---

**Hard Filter 상세:**

**H1. 성별 불일치**

| 항목 | 내용 |
|------|------|
| 목적 | 사용자가 온보딩에서 선택한 성별에 맞는 상품만 코디에 포함 |
| 로직 | 상품의 `gender` 속성이 사용자 성별과 일치하거나 `unisex`이면 통과 |
| 성별 분류 | 하이브리드 3단계 — 키워드(~50%) → 카테고리 추론(~30%) → LLM 캐시(~20%). 5.4.1 체계와 동일 구조로, 기존 LLM 분류 요청에 `gender` 필드를 추가하여 한 번의 요청으로 동시 처리한다 |

```
사용자 성별 = female → 통과: female / unisex, 제거: male
사용자 성별 = male   → 통과: male / unisex, 제거: female
```

**H2. 예산 초과**

| 항목 | 내용 |
|------|------|
| 목적 | 사용자 예산을 크게 벗어난 코디 제거 |
| 로직 | 코디 총액 > 사용자 예산 상한 × 1.5 이면 제거 |
| 왜 1.5배 | 약간의 초과(10~30%)는 Soft Score(PE)에서 감점으로 처리. 50% 이상 초과는 "추천 의미 없음"으로 판단 |
| 예산 미만 | Hard Filter 아님. 저가 코디는 PE에서 약간 감점만 (가성비 관점에서 오히려 긍정적일 수 있음) |

**H3. 계절 완전 불일치**

| 항목 | 내용 |
|------|------|
| 목적 | 현재 시즌과 완전히 맞지 않는 코디 제거 |
| 로직 | 코디 태그의 시즌과 현재 월을 비교. 완전 반대 시즌(예: 겨울 코트 → 7~8월) 제거 |
| 시즌 매핑 | 3~5월: spring, 6~8월: summer, 9~11월: autumn, 12~2월: winter |
| 인접 시즌 | 허용. 봄 코디를 초여름에 보여주거나, 가을 코디를 늦여름에 보여주는 것은 OK |
| 적용 제외 | TPO가 "여행"인 경우 시즌 필터 완화 (여행지의 기후가 다를 수 있음) |

**H4. TPO 완전 불일치**

| 항목 | 내용 |
|------|------|
| 목적 | 사용자 TPO와 코디의 designed_tpo가 완전히 다른 경우 제거 |
| 로직 | TPO 동의어 확장(5.5.2) 적용 후에도 매칭 0이면 제거 |
| 예시 | 면접 사용자에게 workout/travel 코디 → 제거 |
| "전체" 탭 | TPO 탭이 "전체"일 때는 H4 비적용. 사용자의 온보딩 TPO 중 1개라도 매칭되면 통과 |

**H5. 브랜드 화이트리스트**

| 항목 | 내용 |
|------|------|
| 목적 | 인지도 낮은 브랜드 상품 제거로 추천 신뢰도 확보 |
| 로직 | 코디 내 아이템 중 1개 이상이 화이트리스트 브랜드면 통과 |
| 화이트리스트 규모 | 약 120개 이상 (무신사 스탠다드, 유니클로, COS 등) |
| 왜 "1개 이상" | 전체가 인지 브랜드일 필요 없음. 최소 품질 앵커를 보장하면서 가격 효율 조합 허용 |

**H6. LLM 품질**

| 항목 | 내용 |
|------|------|
| 목적 | 규칙으로 잡지 못하는 미묘한 스타일링 부조화 제거 |
| 로직 | Gemini Flash 5점 척도 평가, 3점 미만 제거 |
| 비용 통제 | H1~H5를 먼저 적용하여 LLM 평가 대상 최소화 |

**H7. 톤 호환성** *(P1 우선 원칙 적용)*

| 항목 | 내용 |
|------|------|
| 목적 | 사용자 퍼스널컬러 톤과 완전 불일치 코디 제거 |
| 로직 | 사용자 톤 + 호환 톤 집합에 매칭되는 아이템이 0개이면 제거 |
| 왜 0개 기준 | 1개라도 매칭되면 포인트 아이템으로 활용 가능. 완전 불일치만 제거 |
| **P1 우선 원칙** | 퍼스널컬러는 가이드이지 제한이 아니다. H7은 기술적 안전망(겨울쿨딥에 봄웜라이트 전용 파스텔 등 극단적 불일치만 제거)일 뿐, 사용자의 선택지를 과도하게 제한하지 않는다. H7 필터율이 30%를 초과하면 호환 톤 집합을 확장하여 조정한다. Soft Score PCF 축에서 톤 거리가 자연스럽게 점수에 반영되므로, Hard Filter는 최소한으로 운용한다 |

**H8. StyleFilter 컷오프**

| 항목 | 내용 |
|------|------|
| 목적 | 카테고리 궁합·실루엣·포멀도가 명백히 부조화한 코디 제거 |
| 로직 | StyleFilter 3축 가중합 55점 미만 제거 |
| 근거 | QA 검증(6.5.1)에서 후드+정장+힐 등 부조화 코디를 걸러내기 위해 설정 |

---

**Hard Filter 적용 순서:**

비용이 낮은 순서대로 적용하여 비용이 높은 필터(H6 LLM)의 처리량을 최소화한다.

```
H1 성별 (< 1ms, $0) → H2 예산 (< 1ms, $0) → H3 계절 (< 1ms, $0)
→ H4 TPO (< 1ms, $0) → H5 브랜드 (< 1ms, $0) → H7 톤 (< 1ms, $0)
→ H8 StyleFilter (< 5ms, $0) → H6 LLM (배치, ~$2.5)
```

---

**Soft Score 상세 → 섹션 5.5 (5축 스코어링) 참조**

Hard Filter를 통과한 코디에 대해 5축 Soft Score를 계산한다. Soft Score는 코디를 **제거하지 않고 순위만 조정**한다. 이전에 Hard Filter 역할을 하던 항목(예산 초과, TPO 불일치)이 Hard로 분리되었으므로, Soft Score의 축간 패널티는 제거한다.

| Soft Score 축 | 가중치 | 역할 | Hard Filter와의 관계 |
|--------------|--------|------|---------------------|
| PCF 퍼스널컬러 | 0.25 | 톤 적합도 순위 | H7에서 완전 불일치는 이미 제거됨. 남은 코디 내 "더 잘 맞는 정도" |
| OF TPO 적합도 | 0.20 | 상황 매칭 순위 | H4에서 완전 불일치 제거됨. 남은 코디 내 매칭 비율 |
| CH 색상 조화 | 0.15 | 색 조합 순위 | Hard 대상 아님. 조화도는 상대적 |
| PE 가격 효율 | 0.15 | 예산 효율 순위 | H2에서 50%+ 초과 제거됨. 남은 범위 내 효율 |
| SF 스타일 핏 | 0.25 | 스타일 순위 | H8에서 55점 미만 제거됨. 남은 코디 내 스타일 품질 |

#### 색상 → 퍼스널컬러 톤 매핑 알고리즘

상품 이미지에서 추출한 대표 색상(HEX)을 12-tone 중 가장 가까운 톤에 매핑한다.

**데이터 구조:**
- 12-tone 팔레트 DB: 각 톤별 20~30개 대표 RGB 색상 배열
- 예: `spring_warm_light → [(255,183,150), (255,213,166), ...]`

**매핑 알고리즘:**

```
Input:  HEX 색상 코드 (예: "#B0A6C6")
Output: 가장 가까운 톤 ID (예: "summer_cool_soft")

1. HEX → RGB 변환
2. 12개 톤 각각에 대해:
   a. 해당 톤의 모든 대표 색상과 RGB 유클리드 거리 계산
   b. 최소 거리를 해당 톤의 대표 거리로 채택
3. 12개 톤 중 대표 거리가 가장 작은 톤을 매핑 결과로 반환
```

**왜 RGB 유클리드 거리인가:**
- CIE LAB 색공간 기반의 Delta E가 인간의 색 지각에 더 정확하지만, RGB → LAB 변환에 illuminant 보정이 필요하고 연산 비용이 높다.
- 패션 상품의 색상은 대부분 고채도/중채도 영역에 분포하며, 이 영역에서 RGB 유클리드 거리와 CIE76(Delta E)의 상관계수가 약 0.85 이상으로 실용적 정확도를 보인다.
- 25,000개 이상 상품에 대해 반복 계산하므로, 연산 효율이 중요하다. 향후 CIE LAB Delta E 기반으로 업그레이드를 검토한다.

**톤 적합도 계산:**

```
Input:  상품 톤 ID, 사용자 톤 ID
Output: 적합도 (0~100)

규칙:
  동일 톤             → 100점
  호환 톤 (같은 시즌)  → 80~95점 (동일 시즌 내 유사도에 따라)
  반대 시즌 (웜↔쿨)   → 40~60점 (명도/채도 유사도에 따라)
  기본값 (매핑 없음)   → 40점

12 × 12 = 144개 조합에 대해 전문가 자문 기반의 호환성 매트릭스를 사전 정의한다.
```

### 5.5 코디 스코어링 알고리즘

5축 스코어링 체계를 적용한다. 기존 4축(PCF/OF/CH/PE)에 Style Fit(SF) 축을 추가하여 아이템 간 스타일 조화를 정량 평가하고, 가중치를 재배분한다.

#### 5축 가중치 설계

| 축 | 약칭 | 가중치 | 측정 대상 |
|----|------|--------|----------|
| Personal Color Fit | PCF | **0.25** | 사용자 퍼스널컬러와 코디 색상의 조화 |
| Occasion Fit | OF | **0.20** | 사용자 TPO와 코디 태그의 일치도 |
| Color Harmony | CH | **0.15** | 코디 내 아이템 간 색상 조화 |
| Price Efficiency | PE | **0.15** | 예산 대비 가격 효율성 |
| Style Fit | SF | **0.25** | 카테고리 궁합 + 실루엣 밸런스 + 포멀도 일관성 |

**가중치 설정 근거:**

- **PCF 0.25 (공동 최고 가중치):** ColorFit의 핵심 가치 제안이 "퍼스널컬러 기반 추천"이므로, 색상 적합도에 높은 가중치를 부여한다. 4축 시절의 0.35에서 0.25로 낮춘 것은 SF 축 추가로 재배분한 결과다.
- **SF 0.25 (공동 최고 가중치):** 색상이 맞아도 카테고리 조합이 부자연스러운 코디(예: 패딩 + 미니스커트)는 사용자 경험을 크게 해친다. 스타일 조화를 독립 축으로 분리하여 이를 방지한다.
- **OF 0.20:** TPO 일치는 사용자의 실제 착용 시나리오와 직결되므로 높은 가중치를 유지한다.
- **CH, PE 각 0.15:** 색상 조화와 가격 효율은 보조 지표로, 핵심 축(PCF, SF, OF)보다 낮은 가중치를 부여한다.
- **합계 = 1.00:** 가중합이 0~100 범위를 유지하도록 정규화한다.

**총점 계산 공식:**

```
Total = PCF × 0.25 + OF × 0.20 + CH × 0.15 + PE × 0.15 + SF × 0.25
```

각 축은 0~100 범위의 점수를 산출하며, Total도 0~100 범위가 된다. 개인화 가중치 오버라이드가 있는 경우 해당 가중치로 대체하되, 합이 1.0이 되도록 정규화한다.

---

#### 5.5.1 PCF (Personal Color Fit) — 퍼스널컬러 적합도

**측정 대상:** 코디 아이템들의 색상이 사용자의 12-tone 퍼스널컬러와 얼마나 잘 어울리는지 평가한다.

**계산 방식:** RGB 유클리드 거리 기반 연속 스코어

```
Input:
  - item_tone_ids: 코디 아이템들의 톤 ID 리스트
  - item_hex_colors: 코디 아이템들의 HEX 색상 리스트
  - user_tone_id: 사용자의 퍼스널컬러 톤 ID

계산 로직:
  1단계 - 톤 레벨 매칭 (이산적 판단)
    동일 톤        → 100점
    호환 톤        → 95점
    (호환 톤은 퍼스널컬러 전문가 자문 기반 사전 정의)

  2단계 - 색상 레벨 매칭 (연속적 거리)
    톤 매칭이 안 되는 아이템에 대해:
    a. 사용자 톤의 대표 팔레트(20~30개 RGB) 로드
    b. 아이템 HEX → RGB 변환
    c. 팔레트 내 모든 색상과의 유클리드 거리 중 최소값(d_min) 산출
       d_min = min(sqrt((R1-R2)² + (G1-G2)² + (B1-B2)²))
    d. 거리 → 점수 변환:
       score = max(0, 100 - (d_min / 4.42))
       (4.42 = 최대 RGB 거리 441.67 / 100, 선형 역변환)

  3단계 - 코디 전체 PCF
    모든 아이템의 PCF 점수를 평균하여 코디 수준 PCF를 산출한다.
    PCF = mean(item_pcf_scores)

Output: 0~100 (연속값)
```

**왜 유클리드 거리인가 (CIE LAB Delta E와의 비교):**

| 비교 항목 | RGB 유클리드 | CIE LAB Delta E |
|----------|-------------|-----------------|
| 인간 색 지각 정확도 | 중간 (저채도 영역에서 왜곡) | 높음 (지각 균등) |
| 연산 비용 | O(1) 단순 연산 | O(1)이지만 RGB→XYZ→LAB 변환 필요 |
| 구현 복잡도 | 낮음 | 중간 (illuminant D65 보정 등) |
| 패션 색상 영역 정확도 | **높음** (대부분 중~고채도) | 높음 |

패션 상품의 색상은 대부분 중~고채도 영역에 분포하며, 이 영역에서 RGB 유클리드 거리의 실용적 정확도는 충분하다. 초기에는 구현 속도를 우선하여 RGB 유클리드를 채택하고, 향후 CIE LAB Delta E로 업그레이드를 검토한다.

**임계값 설정 근거:**
- `d_min / 4.42` 변환: RGB 공간에서 최대 거리 sqrt(255² × 3) = 441.67을 100으로 나누어, 거리 0 → 100점, 거리 441.67 → 0점의 선형 매핑을 만든다.
- 호환 톤 95점: 동일 시즌 내 유사 톤은 퍼스널컬러 이론상 80% 이상 호환된다. 100점에 가깝지만 동일 톤 대비 약간 감점하여 정확한 톤 매칭을 우대한다.

---

#### 5.5.2 OF (Occasion Fit) — TPO 적합도

**측정 대상:** 코디의 용도/상황 태그가 사용자가 설정한 TPO(Time/Place/Occasion)와 얼마나 일치하는지 평가한다.

**TPO 동의어 확장 설계:**

패션에서는 동일한 상황을 다양한 용어로 표현한다. 예를 들어 "출근룩"과 "오피스룩"은 사실상 같은 TPO이다. 동의어 확장 없이 단순 문자열 매칭하면 recall이 크게 떨어진다.

```
동의어 매핑:
  "commute"   → {"office", "commute"}
  "office"    → {"office", "commute"}
  "weekend"   → {"casual", "weekend", "daily"}
  "casual"    → {"casual", "weekend", "daily"}
  "daily"     → {"casual", "daily", "weekend"}
  "interview" → {"interview", "office"}
  "campus"    → {"campus", "casual"}
  "event"     → {"party", "wedding", "event"}
  "party"     → {"party", "event"}
  "wedding"   → {"wedding", "event"}
  "workout"   → {"workout"}
```

**매핑 설계 근거:**
- **commute ↔ office:** 동일 격식 수준, 아이템 교집합이 90% 이상
- **weekend ↔ casual ↔ daily:** 일상적 착장으로 스타일 스펙트럼이 겹침
- **interview → office (단방향 확장):** 면접은 오피스 복장의 부분 집합이지만, 오피스가 항상 면접에 적합하진 않음. 양방향으로 확장하지 않는다
- **workout:** 운동복은 다른 TPO와 교집합이 없으므로 동의어 없음

**계산 로직:**

```
Input:
  - outfit_tags: 코디에 부여된 TPO 태그 리스트
  - user_tpo_list: 사용자가 설정한 TPO 리스트

계산:
  1. 사용자 TPO를 동의어 확장하여 expanded_tpos 집합 생성
     expanded_tpos = union(synonym_map[tpo] for tpo in user_tpo_list)
  2. outfit_tags와 expanded_tpos의 교집합 크기(match_count) 산출
  3. 매칭 비율 → 점수 변환:
     match_count >= 2  → 80 + (match_count / total_tags) × 20  (최대 100)
     match_count == 1  → 60 + (1 / total_tags) × 20
     match_count == 0  → 30점 (기본점, 완전 미매칭)

Output: 0~100 (30점 하한)
```

**왜 0이 아닌 30점 하한인가:**
- TPO 태그가 불완전할 수 있다(태깅 누락). 완전 미매칭이라고 0점을 주면 태깅 품질에 따라 점수 변동이 과도해진다.
- 30점은 "확인 불가"를 의미하며, 다른 축(PCF, CH, SF)의 점수가 높으면 여전히 추천 대상이 될 수 있도록 한다.

---

#### 5.5.3 CH (Color Harmony) — 색상 조화도

**측정 대상:** 코디 내 아이템들 간의 색상 조합이 시각적으로 조화로운지 평가한다.

**이론적 기반 — 60-30-10 법칙:**

인테리어/패션 디자인의 60-30-10 법칙에 따르면, 조화로운 색상 구성은 다음 비율을 따른다:
- 60% 메인 컬러 (base, 안정감)
- 30% 서브 컬러 (보조, 깊이)
- 10% 포인트 컬러 (강조, 생동감)

이 법칙의 핵심은 **색상 간 적절한 거리**다. 너무 유사하면 단조롭고, 너무 다르면 산만하다.

**계산 로직:**

```
Input: item_hex_colors (코디 아이템들의 HEX 색상 리스트)

계산:
  1. 모든 아이템 쌍의 RGB 유클리드 거리(d) 계산
  2. 평균 거리(d_avg)를 기반으로 구간별 점수 산출:

     d_avg < 30         → 60점 (너무 유사, 단조로움)
     30 ≤ d_avg < 80   → 80 + (d_avg - 30) / 50 × 20 (유사색 조화, 80~100점)
     80 ≤ d_avg < 150  → 100 - (d_avg - 80) / 70 × 21 (적절한 대비, 79~100점)
     d_avg ≥ 150       → max(30, 79 - (d_avg - 150) / 290 × 49) (과도한 대비, 30~79점)

  3. 채도 보너스:
     아이템이 3개 이상이고, 채도(Saturation) 표준편차가 0.15~0.40이면 +5점
     (적절한 채도 대비가 시각적 생동감을 더한다)

Output: 0~100
```

**구간별 임계값 설정 근거:**

| 거리 구간 | 색상 관계 | 점수 범위 | 패션 해석 |
|----------|----------|----------|----------|
| 0~30 | 거의 동일색 | 60점 | 올 블랙/올 화이트는 의도적 선택일 수 있으나, 일반적으로 단조로움 |
| 30~80 | 유사색(Analogous) | 80~100점 | 톤온톤, 그라데이션 코디. 안정감 있고 세련된 조합 |
| 80~150 | 보색에 가까운 대비 | 79~100점 | 포인트 컬러 활용. 60-30-10 법칙에 가장 부합 |
| 150+ | 극단적 대비 | 30~79점 | 형광 + 파스텔 등 부자연스러운 조합. 거리가 클수록 감점 |

- RGB 거리 30: 육안으로 "거의 같은 색"으로 인식하는 경계. 예: #FF6B6B와 #FF8888 (거리 ~29)
- RGB 거리 80: 명확히 다른 색이지만 조화로운 범위의 시작. 예: 코랄 핑크와 아이보리 (거리 ~85)
- RGB 거리 150: 보색 관계에 가까운 대비. 예: 네이비와 오렌지 (거리 ~160)

**채도 보너스 조건 (표준편차 0.15~0.40):**
- 0.15 미만: 모든 아이템의 채도가 비슷 → 보너스 불필요 (이미 유사색 구간에서 높은 점수)
- 0.15~0.40: 적절한 채도 변화 → 시각적 깊이를 더하므로 +5점
- 0.40 초과: 채도 차이가 과도 → 부자연스러운 조합 가능성, 보너스 미부여

---

#### 5.5.4 PE (Price Efficiency) — 가격 효율성

**측정 대상:** 코디의 총 가격이 사용자가 설정한 예산 범위 대비 얼마나 효율적인지 평가한다.

**계산 로직:**

```
Input:
  - total_price: 코디 총 가격
  - budget_min: 사용자 최소 예산
  - budget_max: 사용자 최대 예산

계산:
  budget_mid = (budget_min + budget_max) / 2

  Case 1: budget_min ≤ total_price ≤ budget_max (예산 범위 내)
    → 100 - |total_price - budget_mid| / budget_mid × 30
    (예산 중앙에 가까울수록 높은 점수, 최대 100점)

  Case 2: total_price > budget_max (예산 초과)
    over_ratio = (total_price - budget_max) / budget_max
    → max(0, 70 - over_ratio × 100)
    (10% 초과 → 60점, 50% 초과 → 20점, 70% 초과 → 0점)

  Case 3: total_price < budget_min (예산 미만)
    under_ratio = (budget_min - total_price) / budget_min
    → max(40, 80 - under_ratio × 80)
    (예산 미만도 감점, 최저 40점)

Output: 0~100
```

**왜 예산 미만도 감점하는가:**

예산보다 너무 저렴한 상품은 **품질 의심** 신호다. 사용자가 10만원 예산을 설정했는데 2만원 코디를 추천하면, "저품질 상품 추천"이라는 부정적 인식이 생긴다. 다만 예산 초과보다는 감점 폭을 작게 설정한다(최저 40점 vs 최저 0점).

**예산 효율 곡선:**

```
점수
100 ─ ─ ─ ─ ─ ─ ─ ┐
                    │   예산 범위 내: 중앙이 최고점
```

#### 5.5.5 SF (Style Fit) — 스타일 적합도

**측정 대상:** 코디 아이템들 간의 카테고리 궁합, 실루엣 밸런스, 포멀도 일관성을 종합 평가한다. 세부 설계는 섹션 6.6에서 상세히 다룬다.

**계산 로직:**

```
Input: 코디 아이템 리스트 (각 아이템의 title, category3)

계산:
  1. 카테고리 궁합 점수 (가중치 50%)
     → 227개 카테고리 조합 매트릭스에서 점수 조회
  2. 실루엣 밸런스 점수 (가중치 25%)
     → Y/A/I/X 라인 조합 규칙에서 점수 조회
  3. 포멀도 일관성 점수 (가중치 25%)
     → 아이템 간 포멀도(1~5) 편차 기반 감점

  SF = category_score × 0.50 + silhouette_score × 0.25 + formality_score × 0.25

Output: 0~100
```

**55점 컷오프 근거:**

SF 점수가 55점 미만인 코디는 피드에서 사전 제거(StyleFilter)한다.
- 초기 45점에서 QA 검증(6.5.1 참조)을 통해 55점으로 상향 조정한다. 후드+정장 등 명백한 부조화 코디의 필터링을 강화하기 위함이다.
- 55점은 3요소(카테고리 50%, 실루엣 25%, 포멀도 25%) 중 **두 요소 이상이 보통 이하이면** 탈락하는 수준이다.
- 이 컷오프는 "명백히 이상한 조합"을 확실히 제거하면서도, 경계선의 창의적 조합은 허용하는 수준이다.

## 6. 추천 엔진 상세 설계

### 6.1 코디 추천 파이프라인

코디 추천 파이프라인은 7단계로 구성한다. StyleFilter(3단계)와 GeminiStyleScorer(6단계)를 포함하여 규칙 기반 사전 필터와 AI 기반 품질 보증 계층을 갖춘다.

| 단계 | 입력 | 처리 | 출력 | 성능 요구사항 |
|------|------|------|------|-------------|
| 1. Profile Load | 사용자 ID | DB에서 프로필 조회. 퍼스널컬러 톤, TPO 리스트, 예산 범위, 개인화 가중치(있으면) 로드 | UserProfile + weight_overrides | < 50ms (DB 인덱스 활용) |
| 2. Candidate Filter | 전체 코디 DB | 톤 호환성 필터 + 브랜드 화이트리스트 + 성별 필터 + LLM 품질 점수(3점+) 적용. 명백히 부적합한 코디 제거 | 후보 코디 리스트 | < 200ms (메모리 필터링) |
| 3. StyleFilter | 후보 코디 | 규칙 기반 사전 필터. 카테고리 궁합(50%) + 실루엣 밸런스(25%) + 포멀도 일관성(25%) 합산. 55점 미만 제거 | 통과 코디 + style_rule_score | < 100ms (룩업 테이블) |
| 4. Scoring | 통과 코디 | 5축 스코어링 (PCF 0.25 + OF 0.20 + CH 0.15 + PE 0.15 + SF 0.25). 개인화 가중치 오버라이드 적용 | ScoredOutfit 리스트 | < 300ms (수식 연산) |
| 5. Re-ranking | 점수 코디 | 완성도 가산(상하의+아우터 → +3점) + dislike 제외 + 톤 다양성(동일 톤 3개 제한) + 메인아이템 중복 제거(1개 제한) + 개인화 보정(-10~+10) | 상위 200개 | < 100ms (정렬) |
| 6. GeminiStyleScorer | 상위 20개 | Gemini 멀티모달 AI가 이미지 기반으로 스타일 평가. 선택적 적용(style_scoring=true). SF 점수를 AI 평가 점수로 갱신 후 재정렬 | SF 갱신 + 재정렬 | < 5s (API 호출, 비동기) |
| 7. Reason Gen | 페이지 코디 | 5축 가중 기여도 상위 2개 축을 선정하여 자연어 추천 이유 생성 | 코디 + 이유 2줄 | < 50ms (템플릿 매칭) |

**전체 파이프라인 응답 시간 목표:**
- GeminiStyleScorer 미적용 시: < 800ms (1~5단계 + 7단계)
- GeminiStyleScorer 적용 시: < 6s (전체 7단계, Gemini API 병렬 호출)

**설계 원칙 — 깔때기(Funnel) 구조:**

비용이 낮은 필터를 먼저 적용하여 후속 단계의 처리량을 줄인다.
```
전체 코디 DB (~1,000개)
  → [2. 톤/브랜드/성별/LLM 필터] → ~500개
    → [3. StyleFilter] → ~300개
      → [4. 5축 스코어링] → ~300개 (점수 부여)
        → [5. 리랭킹] → 200개
          → [6. Gemini AI] → 상위 20개만 (선택적)
            → [7. 이유 생성] → 페이지 단위 (10~20개)
```

### 6.2 유사 상품(Similar) 매칭

코디 내 특정 아이템에 대해 색상과 가격이 유사한 대안 상품을 찾아 제시한다. "Exact"(동일 상품 다른 판매처)와 "Similar"(유사 색상/카테고리 대체재)를 구분한다.

**유사도 산출 공식:**

```
Input:
  - 기준 상품 (category, color_hex, price, name, brand)
  - 후보 상품 풀 (동일 카테고리)

유사도 계산:
  color_similarity = 1.0 - euclidean_distance(rgb1, rgb2) / 441.67
  price_similarity = min(price1, price2) / max(price1, price2)

  total_similarity = color_similarity × 0.6 + price_similarity × 0.4

매칭 유형 판별:
  name_normalized 동일 + brand 동일 → "exact" (동일 상품 다른 판매처)
  그 외 → "similar" (대체재)

Output: 유사도 내림차순 상위 5개
```

**가중치 근거:**
- 색상 유사도 0.6: ColorFit의 핵심 가치가 "색상 매칭"이므로 색상 유사성에 더 높은 가중치를 부여한다.
- 가격 유사도 0.4: 대안 상품의 가격이 원래 상품과 크게 다르면 사용자 기대에 맞지 않는다.

### 6.3 Top Pick 단일 추천

사용자에게 "가장 잘 어울리는 1개 코디"를 선정하여 강력 추천한다. 결정 피로(decision fatigue)를 해소하는 핵심 기능이다.

**선정 로직:**

```
1. 저장 목록 기반 (saved outfits 존재 시)
   사용자가 저장한 코디 목록을 대상으로 실시간 5축 스코어링을 재수행한다.
   최고 점수 코디를 Top Pick으로 선정한다.
   → 사용자가 이미 관심을 표현한 코디 중에서 선택하므로 만족도가 높다.

2. 전체 DB 기반 (저장 목록 없을 시)
   전체 코디 DB에서 5축 스코어링 상위 코디를 선정한다.
   → 콜드스타트 사용자에게도 추천을 제공한다.
```

**시간대 기반 One-shot 추천:**

현재 시간대를 기반으로 TPO를 자동 추론하여, 상황에 맞는 코디 3개를 즉시 추천한다.

```
시간대 → TPO 매핑:
  오전 (6~12시)  → ["office", "interview", "daily"]  (출근 시간)
  오후 (12~18시) → ["casual", "travel", "daily"]     (활동 시간)
  저녁 (18~6시)  → ["date", "party", "daily"]        (저녁 약속)

사용자 설정 TPO + 시간대 TPO를 합산하여 스코어링에 반영한다.
```

**A/B 비교 기능:**

두 코디를 5축 기준으로 비교하여 어느 쪽이 더 적합한지 판정한다. 축별 점수 차이 중 가장 큰 항목을 "결정적 차이 요인"으로 제시한다.

### 6.4 추천 이유 생성

5축 기반 이유 생성을 설계한다. SF(Style Fit) 축 템플릿을 포함하고, 점수 구간별 high/mid 분기를 적용한다.

**설계 원칙:**
1. 사용자는 "왜 이 코디가 추천되었는지" 알고 싶어한다.
2. 5축 중 **가중 기여도가 가장 높은 2개 축**을 선정하여 자연어로 변환한다.
3. 기여도 = 해당 축 점수 × 해당 축 가중치 (점수가 높고 가중치도 높은 축이 선택됨)

**이유 선정 알고리즘:**

```
Input: ScoredOutfit (pcf, of, ch, pe, sf 점수 + 가중치)

1. 가중 기여도 계산:
   contributions = [
     ("pcf", pcf × 0.25),
     ("of",  of  × 0.20),
     ("ch",  ch  × 0.15),
     ("pe",  pe  × 0.15),
     ("sf",  sf  × 0.25),
   ]

2. 기여도 내림차순 정렬 → 상위 2개 축 선택

3. 각 축의 raw 점수에 따라 템플릿 분기:
   raw_score >= 75 → "high" 템플릿 (적극적 표현)
   raw_score < 75  → "mid" 템플릿 (완화된 표현)

Output: 2줄의 자연어 추천 이유
```

**템플릿 구조:**

| 축 | 레벨 | 예시 템플릿 |
|----|------|-----------|
| pcf | high | "{톤이름} 핵심 컬러와 잘 어울려서 피부톤이 한층 밝아 보여요" |
| pcf | mid | "퍼스널컬러와 비교적 잘 어울리는 색상 구성이에요" |
| of | high | "{TPO} 룩에 적합한 스타일링이에요" |
| of | mid | "다양한 상황에 무난하게 활용할 수 있는 스타일이에요" |
| ch | high | "메인-서브-포인트 컬러가 균형 있게 조화를 이뤄요" |
| ch | mid | "전체적으로 안정감 있는 색상 배합이에요" |
| pe | high | "예산 범위 내에서 가성비 좋은 조합이에요" |
| pe | mid | "가격 대비 만족스러운 구성이에요" |
| sf | high | "스타일 조화가 뛰어난 코디예요" / "실루엣 밸런스가 좋은 세련된 코디예요" |
| sf | mid | "전체적으로 무난한 스타일 구성이에요" |

**75점 분기 기준 근거:**
- 75점 이상: 해당 축에서 "상위 25%" 수준으로, 적극적으로 장점을 어필할 수 있다.
- 75점 미만: 장점이 약하므로 완화된 표현으로 부정적 인상을 피한다.
- 각 축 점수 분포의 3사분위수(Q3)가 대체로 70~80점 부근이므로, 75점이 자연스러운 경계가 된다.

### 6.5.1 알고리즘 검증 (QA)

8개 시나리오 기반 검증을 수행하여 Hard Filter와 Soft Score의 정합성을 확인한다.

**검증 시나리오 및 결과:**

| # | 시나리오 | Hard Filter | Soft Score | 최종 판정 |
|---|---------|-------------|------------|----------|
| 1 | ✅ 봄웜라이트 면접 (블라우스+슬랙스+로퍼) | 전체 통과 | 91.2 | 🟢 추천 |

| 3 | ⚠️ 톤 완전 불일치 (겨울딥→봄라이트) | **H7 톤 호환 0매칭 → 제거** | — | 🔴 Hard 탈락 |

**Hard/Soft 분리의 효과:**

| | 기존 (패널티 방식) | 변경 (Hard/Soft 분리) |
|---|---|---|
| 예산 67% 초과 | PE 23점 + 패널티 -10 = **68점으로 피드에 남음** | H2 → **리스트에서 제거** |
| 파티→면접 TPO | OF 30점 + 패널티 -10 = **67점으로 피드에 남음** | H4 → **리스트에서 제거** |
| 후드+정장+힐 | SF 65점 + 패널티 -10 = **61점으로 피드에 남음** | H8 → **리스트에서 제거** |
| 좋은 코디(블라우스+슬랙스) | 91점 (패널티 없음) | Hard 통과 → Soft 91점 (**동일**) |

### 6.6 스타일 호환성 매트릭스 (StyleFilter)

규칙 기반 사전 필터로, 명백히 부조화한 코디를 스코어링 전에 빠르게 제거한다. 카테고리 감지에는 5.4.1 하이브리드 분류 체계(키워드 + LLM 폴백)를 사용하여, 카테고리 감지율을 97%로 확보한다.

#### 카테고리 감지 → 하이브리드 분류 체계 연동

StyleFilter의 `detect_category()` 는 단순 키워드 매칭에 의존하지 않는다. 5.4.1에서 정의한 2단계 폴백 구조를 따른다:

1. **키워드 매칭** (~70% 커버) — 31종 카테고리 × 3~5개 키워드 딕셔너리 룩업. < 1ms.
2. **LLM 분류 캐시 조회** (~27% 커버) — 배치 사전 분류 결과. product_id로 즉시 조회. < 1ms.
3. **미분류 폴백** (~3%) — 기본값 적용 또는 런타임 LLM 호출.

이 구조에 의해 StyleFilter의 카테고리 궁합·실루엣·포멀도 판단이 정확해지며, 227개 궁합 규칙의 입력 품질이 보장된다.

#### 3축 구성 원리

StyleFilter는 패션 스타일링의 3가지 기본 원칙을 정량화한다:

**1. 카테고리 궁합 (50%)**

아이템 카테고리 간 조합의 자연스러움을 평가한다. 227개 조합 규칙으로 구성된다.

```
조합 카테고리:
  상의-하의:   블라우스+슬랙스(90), 니트+청바지(85), 셔츠+레깅스(40), ...
  상의-아우터:  블라우스+트렌치(95), 후드티+코트(50), 크롭탑+패딩(30), ...
  하의-신발:   슬랙스+로퍼(90), 청바지+스니커즈(85), 미니스커트+등산화(20), ...
  원피스-아우터: 원피스+가디건(90), 원피스+패딩(60), ...
  기타 조합:   가방+신발 스타일 일치 보너스 등
```

**227개 조합의 구성 원리:**
- 7개 대카테고리(top, bottom, outer, onepiece, shoes, bag, acc) 내 세부 카테고리(블라우스, 니트, 셔츠 등)를 정의한다.
- 세부 카테고리 간 모든 유의미한 2-아이템 조합에 대해 0~100점을 부여한다.
- 점수 기준: 패션 스타일링 가이드 + 쇼핑몰 코디 데이터 분석 기반

**2. 실루엣 밸런스 (25%)**

상하 실루엣 조합의 자연스러움을 패션 이론(Y/A/I/X 라인)에 기반하여 평가한다.

```
실루엣 라인 정의:
  Y 라인: 상체 볼륨 + 하체 슬림 (오버사이즈 상의 + 스키니)
  A 라인: 상체 슬림 + 하체 볼륨 (피트 상의 + 와이드 팬츠)
  I 라인: 상하 균일 (슬림 상의 + 슬림 하의, 또는 오버 + 오버)
  X 라인: 허리 강조 (크롭 + 하이웨이스트, 벨트 포인트)
```

**실루엣 조합 규칙 (15개):**

| 상의 실루엣 | 하의 실루엣 | 라인 | 점수 | 패션 이론 근거 |
|-----------|-----------|------|------|-------------|
| oversized | slim/skinny | Y | 95 | 가장 보편적인 밸런스. 상체 볼륨을 하체 슬림으로 중화 |
| fitted | wide/flared | A | 95 | 여성스러운 실루엣. 상체를 깔끔하게 정리하고 하체로 시선 유도 |
| fitted | slim | I | 85 | 깔끔하고 단정한 인상. 체형 보정 효과는 적으나 모던한 느낌 |
| oversized | wide | I (볼륨) | 60 | 전체적으로 볼륨감 과다. 의도적 스타일링이 아니면 산만함 |
| crop | high_waist | X | 90 | 허리 라인 강조. 비율이 좋아 보이는 효과 |
| ... | ... | ... | ... | ... |

**3. 포멀도 일관성 (25%)**

아이템 간 격식도(formality) 편차를 최소화하여 코디의 통일감을 보장한다.

```
포멀도 스펙트럼 (1~5):
  1 = 스포츠/애슬레저 (레깅스, 운동화, 후드집업)
  2 = 캐주얼 (청바지, 스니커즈, 티셔츠)
  3 = 스마트캐주얼 (치노팬츠, 로퍼, 니트)
  4 = 비즈니스캐주얼 (슬랙스, 블라우스, 더비슈즈)
  5 = 포멀 (정장, 원피스, 하이힐, 클러치)
```

**포멀도 설정 근거:**
- 5단계 스펙트럼은 패션 업계에서 널리 사용되는 Dress Code 분류(Casual → Business Casual → Business → Formal → Black Tie)를 단순화한 것이다.
- 각 아이템에 포멀도 점수를 부여하고(33개 규칙), 코디 내 아이템 간 포멀도 표준편차를 기반으로 점수를 산출한다.

```
포멀도 일관성 계산:
  std_dev = 아이템 포멀도 값들의 표준편차
  score = max(0, 100 - std_dev × 40)

  std_dev = 0 (완전 일치)     → 100점
  std_dev = 0.5 (1단계 편차)  → 80점
  std_dev = 1.0 (2단계 편차)  → 60점
  std_dev = 1.5 (3단계 편차)  → 40점
  std_dev ≥ 2.5              → 0점
```

포멀도 편차가 2단계 이상이면(예: 정장 재킷 + 운동화) 명백히 부조화하므로 강하게 감점한다.

### 6.7 Gemini 멀티모달 스타일 스코어러

AI 기반 스타일 평가 모듈로, 규칙 기반 StyleFilter가 잡지 못하는 미묘한 스타일 부조화를 이미지 분석으로 감지한다.

#### 평가 기준 (5개 세부축)

| 세부축 | 가중치 | 설명 |
|--------|--------|------|
| style_cohesion | 0.30 | 아이템들의 스타일 통일감 (캐주얼끼리, 포멀끼리) |
| silhouette_balance | 0.25 | 상하 실루엣 밸런스 (Y/A/I/X 라인 자연스러움) |
| trend_relevance | 0.15 | 현재 트렌드 적합도 (클린핏, 올드머니, 발레코어 등) |
| material_harmony | 0.15 | 소재 조화 (계절감, 소재 믹스 자연스러움) |
| overall_styling | 0.15 | 전체적인 스타일링 완성도 |

**프롬프트 설계 방향:**
- 코디 아이템 이미지를 멀티모달 입력으로 전달한다.
- 5개 세부축 각각에 대해 1~10점 평가를 요청한다.
- 가중합으로 최종 SF 점수를 산출하여 기존 규칙 기반 SF를 대체한다.

#### 비용 최적화 전략

```
전체 코디 → StyleFilter(규칙, 무료) → 5축 스코어링(수식, 무료) → 리랭킹
                                                                      │
                                                                 상위 20개만
                                                                      │
                                                                      ▼
                                                          GeminiStyleScorer(AI, 유료)
                                                          Gemini 2.0 Flash 모델 사용
                                                          동시 호출 5건 제한
                                                          결과 캐싱 (재요청 시 API 호출 생략)
```

**비용 최적화 설계 근거:**
1. **적용 범위 제한:** 전체 코디가 아닌 리랭킹 후 상위 20개에만 적용. 비용을 1/50 수준으로 절감.
2. **모델 선택:** Gemini 2.0 Flash는 Pro 대비 비용이 1/10이면서 이미지 분석 정확도가 충분하다.
3. **동시 호출 제한:** max_concurrent=5로 API Rate Limit 초과를 방지한다.
4. **결과 캐싱:** 동일 코디에 대한 재평가를 방지. 캐시 유효 기간은 7일로 설정한다.
5. **선택적 적용:** `style_scoring=true` 파라미터로 제어하여, 필요한 경우에만 호출한다.

### 6.8 사용자 피드백 개인화 학습

사용자 피드백을 기반으로 5축 가중치를 동적으로 조정하는 개인화 모듈을 설계한다.

#### 피드백 행동별 신호 가중치

| 행동 | 가중치 | 의미 | 설계 근거 |
|------|--------|------|----------|
| save | +2.0 | 강한 긍정 신호 | "나중에 다시 보겠다"는 의도. 구매 의향에 가장 가까운 행동 |
| like | +1.0 | 긍정 신호 | 호감 표현이지만 save보다 commitment가 낮음 |
| click | +0.3 | 약한 관심 신호 | 관심은 있으나 의도적 선택인지 불분명. 노이즈 방지를 위해 낮은 가중치 |
| dislike | -1.5 | 부정 신호 | 명시적 거부. save(+2.0)보다 절대값이 낮은 이유: 부정 신호 과잉 반영 시 추천 다양성이 급격히 축소됨 |

#### 학습 구조

```
사용자 행동 (save/like/click/dislike)
    ↓
PreferenceTracker에 피드백 기록
    ↓
4가지 선호도 프로필 누적:
  - tone_preferences: {톤 ID → 누적 점수}          ← 어떤 톤을 좋아하는지
  - category_preferences: {카테고리 → 누적 점수}    ← 어떤 카테고리를 좋아하는지
  - brand_preferences: {브랜드 → 누적 점수}         ← 어떤 브랜드를 좋아하는지
  - price_preference: {avg_liked_price, count}      ← 선호 가격대
    ↓
피드백 10건 이상 축적 시 → weight_overrides 자동 생성
    ↓
5축 가중치 동적 조정 (기본 가중치에서 ±0.05 범위)
+ 개인화 보정 점수 (-10 ~ +10)
```

#### 가중치 조정 로직

개인화 가중치는 사용자의 피드백 패턴에서 **선호 집중도**를 분석하여 조정한다:

| 조건 | 조정 | 근거 |
|------|------|------|
| 톤 선호 집중 (긍정 피드백의 80%+가 2개 이하 톤) | PCF 가중치 +0.05 | 톤에 민감한 사용자 → 색상 매칭 중요도 상향 |
| 카테고리 선호 뚜렷 (상위 3개가 80%+) | SF 가중치 +0.05 | 스타일 취향이 명확 → 스타일 적합도 중요도 상향 |
| 가격 선호 패턴 존재 (5건+ 저장의 가격 편차 < 20%) | PE 가중치 +0.03 | 가격 민감 사용자 → 가격 효율 중요도 상향 |
| 전체 가중치 합 = 1.0으로 정규화 | — | 총점 범위 유지 |

**콜드스타트 해결 — 3단계 개인화 진화:**

| 단계 | 조건 | 개인화 소스 | 영향력 |
|------|------|-----------|--------|
| Seed | 피드백 0건 (온보딩 직후) | **Style Seed** (비주얼 취향 분석 4라운드 결과) | SF 축에 +5~10 보정 |
| Hybrid | 피드백 1~29건 | Style Seed + 피드백 학습 블렌딩 | Seed 비중 점진 감소, 피드백 비중 증가 |
| Learned | 피드백 30건+ | 피드백 학습만 | Style Seed 영향력 0, 순수 학습 기반 |

**Seed → Learned 전환 공식:**

```
seed_weight = max(0, 1.0 - (feedback_count / 30))
learned_weight = 1.0 - seed_weight

effective_bonus = seed_bonus × seed_weight + learned_bonus × learned_weight
```

**취향 초기화 시:** Style Seed + PreferenceTracker 데이터를 모두 삭제하고 기본 가중치(0.25/0.20/0.15/0.15/0.25)로 복원한다. 마이페이지에서 재분석(Step 5 재진행) 또는 피드백 초기화를 선택할 수 있다.

**개인화 보정 점수 (-10 ~ +10):**
리랭킹 단계에서 개별 코디에 대해, 사용자의 선호 톤/카테고리/브랜드 일치 여부에 따라 -10 ~ +10점의 보정을 적용한다. 이는 5축 스코어링과 별도로, "이 사용자가 특히 좋아할 만한 코디"를 상위로 올리는 역할을 한다.

### 8.5 인터랙션 & 모션 설계

#### 모션 원칙
- **의미 있는 모션만**: 장식이 아닌, 맥락 전달과 피드백을 위한 모션
- **300ms 이내**: 대부분의 전환은 200~400ms. 사용자가 "빠르다"고 느끼는 범위
- **spring 물리**: 기계적 ease 대신 자연스러운 spring 곡선 사용 (Framer Motion default spring)

#### 인터랙션 맵

| 인터랙션 | 동작 | 모션 명세 | 구현 |
|---------|------|----------|------|
| 카드 등장 | 피드 스크롤 시 뷰포트 진입 | fadeInUp (y: 30→0, opacity: 0→1), duration 0.4s, stagger 0.1s | Framer Motion `whileInView` |
| 저장 (하트) | 하트 아이콘 탭 | scale 0.8→1.2→1.0 (0.3s) + stroke→fill 색 전환 (Marsala) | Framer Motion `animate` |
| 더블탭 저장 | 카드 더블탭 | 카드 중앙에 큰 하트 오버레이 scale 0→1.4→0 (0.6s) + opacity 1→0 | Framer Motion `AnimatePresence` |
| dislike | 좌 스와이프 또는 X 탭 | translateX 0→-120% + opacity 1→0 (0.3s) + 아래 카드 slideUp | Framer Motion `AnimatePresence` + `layout` |
| 피드→상세 | 카드 탭 | 카드 이미지가 풀블리드로 확대 (Shared Element) | Framer Motion `layoutId` |
| 상세→피드 | 뒤로가기 | 역방향 Shared Element (이미지 축소) | Framer Motion `layoutId` |
| 스코어 바 | 상세 페이지 진입 | width 0%→실제값, ease-out 0.8s, stagger 0.15s (축별 순차) | Framer Motion `animate` |
| 탭 전환 | TPO 필터 탭 변경 | 활성 탭 underline 슬라이드 + 콘텐츠 crossfade 0.2s | Framer Motion `layout` + `AnimatePresence` |
| 풀다운 새로고침 | 피드 상단에서 아래로 당기기 | 커스텀 스피너 회전 + 피드 카드 리로드 fadeIn | 커스텀 훅 |
| 바텀 시트 | 필터, 간이 진단, 가격비교 등 | y: 100%→0, spring (stiffness: 300, damping: 30), 배경 dimming 0.2s | Framer Motion `animate` |
| 토스트 알림 | 저장/dislike 완료 피드백 | 하단에서 slideUp + 2초 유지 + fadeOut 0.3s | 커스텀 컴포넌트 |
| parallax | 코디 상세 스크롤 | 이미지 scale 1.0→0.85, 스크롤 비율 연동 | Framer Motion `useScroll` + `useTransform` |

#### 상태별 피드백 패턴

| 상태 | 시각 피드백 | 모션 |
|------|-----------|------|
| 로딩 중 | 스켈레톤 UI (펄스 애니메이션) | opacity 0.4↔0.7 반복, 1.5s cycle |
| 성공 | 콘텐츠 등장 | fadeInUp 0.4s |
| 실패 | 에러 메시지 + 재시도 버튼 | fadeIn 0.3s |
| 빈 상태 | 일러스트 + 안내 텍스트 + CTA | fadeIn 0.5s |
| 저장 완료 | 토스트 "저장했어요" | slideUp + 2s hold + fadeOut |
| dislike 완료 | 토스트 "관심없음" | slideUp + 2s hold + fadeOut |

```
   → 첫 번째 코디: 차콜 재킷 + 버건디 니트 + 네이비 슬랙스

   → 스와이프 왼쪽 = dislike → 다음 코디
   → 두 번째 코디 마음에 듦 → ♡ 저장
4. [반복] 2개 더 스캔 → 1개 추가 저장
5. [퇴근 후] 저장 목록 진입 → 상세 확인 → 가격 비교 → 구매

총 소요 시간: 3분 이내
```

## 11. 진단(온보딩) 설계 상세

### 11.1 랜딩 페이지

| 요소 | 내용 |
|------|------|
| 헤드라인 | "내 색을 아는 순간, 선택이 쉬워진다" |
| 서브카피 | "어울리는 이유까지, 컬러핏이 골라줄게요" |
| CTA | [무료로 시작하기] — 로그인 없이 바로 온보딩 진입 |
| 하단 소셜 프루프 | "3,000명이 컬러핏으로 선택했어요" |

### 11.2 진단 5 Step

온보딩은 5 Step으로 구성한다. 성별 선택을 최초 단계로 배치하고, 마지막에 **비주얼 취향 분석**을 추가하여 콜드스타트 없이 첫 피드부터 개인화된 추천을 제공한다.

**Step 1: 성별 선택 (~3초)**
- [여성] [남성] 2개 버튼. 텍스트 입력 없음.
- 선택에 따라 이후 Step 3의 TPO·무드 옵션, 수집 상품 필터, 코디 레시피가 분화된다.
- "선택하지 않기" 옵션 없음 — 패션 추천의 필수 맥락이므로 반드시 선택.

**Step 2: 퍼스널컬러 설정 (~30초)**
- 이미 아는 사람 → 12-tone 스와치 그리드에서 1탭 선택
- 모르는 사람 → "잘 모르겠어요" → 2문항 간이 진단
  - Q1: "피부톤에 가장 가까운 이미지를 골라주세요" (봄웜/여름쿨/가을웜/겨울쿨)
  - Q2: "평소 자주 입는 상의 색 계열은?" (베이직/어스톤/파스텔/비비드)
- 선택 후 대표색 팔레트 프리뷰 → "이 색들이 나에게 어울리나요?" 확인

**Step 3: TPO & 스타일 무드 (~15초)**
- TPO 2개 선택 (최대 3개): 버튼 탭 방식, 텍스트 입력 없음
- 스타일 무드 복수 선택: 버튼 탭 방식
- **성별에 따라 TPO·무드 세트가 다르다:**

**여성 TPO 옵션:**

| TPO | 설명 | 코디 특성 |
|-----|------|----------|
| 출근 | 오피스 데일리 | 블라우스, 슬랙스, 로퍼 |
| 데이트 | 로맨틱, 첫인상 | 원피스, 스커트, 힐 |
| 면접 | 포멀, 신뢰감 | 블레이저, 셔츠, 펌프스 |
| 주말 | 편안한 외출 | 니트, 청바지, 스니커즈 |
| 캠퍼스 | 대학 일상 | 맨투맨, 데님, 백팩 |

**Step 4: 예산 (~10초)**
- 듀얼 슬라이더 (min ~ max)
- 또는 빠른 선택: [~3만] [3~5만] [5~10만] [10만+]

**Step 5: 비주얼 취향 분석 (~30초)**

사용자의 스타일 취향을 **말이 아닌 눈으로** 파악한다. 샘플 코디 이미지를 보여주고 직감적으로 선택하게 하여, 첫 피드부터 개인화된 추천을 제공한다.

**구성:**
- 사용자의 성별 + 퍼스널컬러 톤에 맞는 샘플 코디 이미지 12~16장을 2×2 그리드로 4라운드 제시
- 각 라운드에서 4장 중 "가장 마음에 드는 코디"를 **1개만 탭**
- 선택하기 싫으면 "패스" 가능 (강제 아님)
- 총 4라운드 = 최대 4개 취향 시그널 수집

**라운드별 설계:**

| 라운드 | 목적 | 제시하는 4장의 차이점 |
|--------|------|---------------------|
| 1 | 무드 선호 파악 | 같은 TPO, 다른 무드 (캐주얼 vs 미니멀 vs 클래식 vs 스트릿) |
| 2 | 실루엣 선호 파악 | 같은 무드, 다른 실루엣 (슬림 vs 오버핏 vs 와이드 vs 피티드) |
| 3 | 색감 선호 파악 | 같은 실루엣, 다른 색감 (모노톤 vs 파스텔 vs 뉴트럴 vs 대비) |
| 4 | 가격대/브랜드 선호 파악 | 비슷한 스타일, 다른 가격대 (SPA vs 디자이너 vs 유니섹스 vs 편집숍) |

**화면 구성:**
```
"어떤 코디가 마음에 드세요?"

┌──────────┐  ┌──────────┐
│          │  │          │
│ 코디 A   │  │ 코디 B   │
│ (미니멀)  │  │ (클래식)  │
│          │  │          │
└──────────┘  └──────────┘
┌──────────┐  ┌──────────┐
│          │  │          │
│ 코디 C   │  │ 코디 D   │

│          │  │          │
└──────────┘  └──────────┘

            [패스 →]

      ● ● ○ ○  (4라운드 중 2)
```

**선택 결과 → 취향 시드(Style Seed) 생성:**

사용자의 4번 선택으로 초기 취향 프로필을 즉시 생성한다. 이 시드는 PreferenceTracker의 **초기 가중치**로 사용된다.

```
선택 분석:
  라운드 1에서 "미니멀" 선택 → mood_preferences: {minimal: +3.0}
  라운드 2에서 "슬림" 선택   → silhouette_preferences: {slim: +2.0}
  라운드 3에서 "모노톤" 선택  → color_preferences: {monotone: +2.0}
  라운드 4에서 "편집숍" 선택  → price_preferences: {mid_high: +1.0}

→ Style Seed 생성:
  {
    "mood_seed": "minimal",
    "silhouette_seed": "slim",
    "color_seed": "monotone",
    "price_seed": "mid_high",
    "seed_confidence": 4  ← 4라운드 모두 선택한 경우
  }
```

**취향 시드의 추천 반영:**
- Soft Score 계산 시 SF(스타일 핏) 축에 시드 보정을 적용한다
- 예: 무드 시드가 "미니멀"인 사용자 → minimal 무드 코디에 SF +5~10 보정
- 피드백 10건 축적 전까지는 이 시드가 유일한 개인화 신호
- 피드백이 쌓이면 시드의 영향력이 점진적으로 감소한다 (30건 이상에서 0)

**초기화:**
- 마이페이지 → "취향 초기화" 버튼
- 탭 시 확인 다이얼로그: "취향 분석 결과와 피드백 학습 데이터를 모두 초기화할까요?"
  - [초기화] → Style Seed + PreferenceTracker 데이터 삭제 → 기본 가중치 복원
  - [취소]
- 초기화 후 "다시 분석하기" 옵션 제시 → Step 5 재진행 가능
- 취향 시드만 리셋하고 피드백 데이터는 유지하는 옵션도 제공

**왜 "선택"이 "텍스트 입력"보다 나은가:**
- 패션 취향은 언어로 표현하기 어렵다. "미니멀"이 뭔지 사람마다 다르게 해석함
- 이미지를 보고 직감적으로 선택하면 실제 취향과 높은 상관관계를 보임
- 에디토리얼 톤과도 맞음 — 이미지 중심 경험

**왜 4라운드인가:**
- 2라운드: 정보 부족. 우연에 의한 선택 가능
- 4라운드: 무드·실루엣·색감·가격의 4개 차원을 독립 측정 가능
- 6라운드+: 이탈 위험. 온보딩 총 시간이 2분을 넘기면 안 됨

### 12.0 초기 전략: 사용자 획득 우선

MVP 단계에서는 구매 전환율보다 **사용자 획득**에 집중한다. 퍼스널컬러 진단 경험과 코디 저장 기능이 초기 후크. 커머스 전환 최적화는 사용자 기반 확보 후 2차 목표로 전환한다.

```
  ├── 온보딩 5 Step
  ├── 코디 피드 UI
  ├── 코디 상세
  └── 아이템 상세 + 가격비교
```

**실행 순서:** Lane A + B 동시 시작 → 둘 다 완료 후 Lane C + D 동시 시작 → 통합 테스트
**충돌 주의:** Lane C와 D는 API 인터페이스(schemas/)를 공유하므로, 인터페이스를 먼저 정의한 후 병렬 시작

### 14.4 DB 스키마 개요

```sql
-- 사용자
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email VARCHAR(255),
    provider VARCHAR(20),          -- kakao, google
    gender VARCHAR(10),            -- female, male (v1.3.1 추가)
    tone_id VARCHAR(30),           -- summer_cool_soft
    tpo_primary VARCHAR(20),
    tpo_secondary VARCHAR(20),
    tpo_list TEXT[],               -- 최대 3개 TPO (v1.3.1 추가)
    style_moods TEXT[],            -- {casual, minimal}
    budget_min INT,
    budget_max INT,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 취향 시드 (온보딩 Step 5 비주얼 취향 분석 결과, v1.3.1 추가)
CREATE TABLE style_seeds (
    id SERIAL PRIMARY KEY,
    user_id UUID REFERENCES users(id) ON DELETE CASCADE,
    mood_seed VARCHAR(30),         -- minimal, casual, classic 등
    silhouette_seed VARCHAR(30),   -- slim, oversized, wide 등
    color_seed VARCHAR(30),        -- monotone, pastel, contrast 등
    price_seed VARCHAR(30),        -- low, mid, mid_high, high
    seed_confidence INT,           -- 0~4 (라운드 참여 수)
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 사용자 선호도 (PreferenceTracker 데이터, v1.3.1 추가)
CREATE TABLE user_preferences (
    id SERIAL PRIMARY KEY,
    user_id UUID REFERENCES users(id) ON DELETE CASCADE,
    tone_preferences JSONB DEFAULT '{}',       -- {톤ID: 누적 점수}
    category_preferences JSONB DEFAULT '{}',   -- {카테고리: 누적 점수}
    brand_preferences JSONB DEFAULT '{}',      -- {브랜드: 누적 점수}
    avg_liked_price INT,
    feedback_count INT DEFAULT 0,
    weight_overrides JSONB,                    -- 5축 가중치 오버라이드
    updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- 상품
CREATE TABLE products (
    id VARCHAR(50) PRIMARY KEY,
    name VARCHAR(500),
    brand VARCHAR(100),
    category VARCHAR(20),          -- top, bottom, outer, onepiece, shoes, bag
    color_hex VARCHAR(7),
    tone_id VARCHAR(30),
    price INT,
    mall_name VARCHAR(50),
    mall_url TEXT,
    image_url TEXT,
    tags TEXT[],
    gender VARCHAR(10),            -- female, male, unisex (v1.3.1 추가)
    silhouette VARCHAR(20),        -- oversized, slim, fitted, wide, regular (v1.3.1 추가)
    formality SMALLINT,            -- 1~5 (v1.3.1 추가)
    last_observed_at TIMESTAMPTZ
);

-- 코디
CREATE TABLE outfits (
    id VARCHAR(50) PRIMARY KEY,
    item_ids TEXT[],               -- product IDs
    gender VARCHAR(10),            -- female, male (v1.3.1 추가)
    designed_tpo VARCHAR(20),      -- 레시피 기반 설계 TPO (v1.3.1 추가)
    designed_moods TEXT[],         -- 레시피 기반 설계 무드 (v1.3.1 추가)
    total_price INT,
    lowest_total_price INT,
    is_complete_outfit BOOLEAN,
    tags TEXT[],
    scores JSONB,                  -- {pcf, of, ch, pe, sf, total}
    style_details JSONB,           -- StyleFilter 결과 (v1.3.1 추가)
    reasons TEXT[],
    llm_quality_score SMALLINT     -- Gemini 평가 1~5 (v1.3.1 추가)
);

-- 사용자 반응
CREATE TABLE reactions (
    id SERIAL PRIMARY KEY,
    user_id UUID REFERENCES users(id),
    outfit_id VARCHAR(50) REFERENCES outfits(id),
    reaction_type VARCHAR(10),     -- save, dislike, click, purchase
    created_at TIMESTAMPTZ DEFAULT NOW()
);
```

### 17.2 개인정보 보호

- 퍼스널컬러 데이터는 서비스 추천 목적으로만 사용
- 저장/반응 데이터는 사용자 동의 하에 추천 개선에 활용
- 계정 삭제 시 모든 개인 데이터 즉시 삭제
- 소셜 로그인 시 최소 정보만 요청 (이메일, 닉네임)

### 17.3 추천의 투명성

- 모든 코디 추천에 반드시 추천 이유(reasons) 표시

- 광고/제휴 상품은 "광고" 라벨 명시
- 스코어링 기준(**5축** 가중치: PCF 0.25 / OF 0.20 / CH 0.15 / PE 0.15 / SF 0.25) 공개 가능
- 개인화 가중치 조정 시에도 전체 가중치 합 = 1.0 유지

### 17.4 체형/외모 민감성

- "어울리지 않는다"는 부정적 표현 배제
- "이 색상이 더 잘 어울려요"로 긍정적 대안 제시
- 체형 관련 표현 없음 (퍼스널컬러 = 색상 중심)
- 모든 성별/체형에 포용적인 언어 사용
